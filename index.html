Stasis Documentation

Stasis is a privacy-first, fully local screen-time tracker, focus scorer, app blocker, and Telegram remote-control tool for Windows. All data lives in a local SQLite database — nothing leaves your machine unless you explicitly request it.

What Stasis does

CapabilityDetails
Screen-time trackingLogs every foreground app with second-granularity, including window title and (for browsers) the active URL.
Productivity & focus scoringA configurable weighted formula rates each day 0–100 using deep-work time, engagement, flow streaks, and switch/idle penalties.
App limit enforcementSet per-app daily time budgets. Stasis automatically terminates processes once limits are reached, with optional temporary unblocks.
Telegram remote controlReceive boot notifications, capture screenshots, pull activity logs, and lock/shutdown your PC from anywhere.
File system monitoringOptional background watcher for file create/modify/delete events across all drives.
Automatic self-updatesChecks GitHub Releases and installs new versions in the background.

Key design principles

  • Local-first — SQLite database at %LOCALAPPDATA%\Stasis\data\stasis.db. No external servers, no cloud sync.
  • Privacy-aware — Telegram credentials are encrypted at rest with Fernet symmetric encryption. The Flask API binds only to 127.0.0.1.
  • Minimal footprint — A compiled Python backend (~100 MB, standalone .exe) + a Tauri WebView2 shell. No Node or Python runtime required at runtime.
  • Configurable — Categories, ignored processes, limits, and all toggles are editable from the UI or directly in JSON/SQLite.

Quick start

# Clone the repository
git clone https://github.com/arshsisodiya/Stasis.git
cd Stasis

# Run backend only (no frontend UI needed for testing)
python -m venv venv && venv\Scripts\activate
pip install -r requirements.txt
python src/main.py
# API is now available at http://127.0.0.1:7432

Or download the pre-built installer from the Releases page ↗.

Installation

Stasis is distributed as a Windows NSIS installer. It requires no external runtimes — everything is bundled inside the setup EXE.

Installing the pre-built release

  1. Go to the latest release page ↗.
  2. Download Stasis-<version>-setup.exe.
  3. Run the installer. Administrator privileges are required.
  4. If WebView2 Runtime is not present, the installer silently downloads and installs it.
  5. Stasis starts automatically after installation and registers itself in the Windows startup registry.
Note: Some antivirus products may flag the PyInstaller-compiled backend. Add %LOCALAPPDATA%\Stasis\ to your AV exclusions if needed.

First-run setup

On the very first launch, Stasis will:

  • Initialize the SQLite database at %LOCALAPPDATA%\Stasis\data\stasis.db.
  • Set default settings and begin tracking the active foreground application immediately.
  • Show the Loading Screen while the Flask API warms up (usually <2 seconds).

Storage locations

PathContents
%LOCALAPPDATA%\Stasis\data\stasis.dbSQLite database (all tracking data, settings, limits)
%LOCALAPPDATA%\Stasis\logs\stasis_YYYY-MM-DD.logDaily application logs (last 7 days kept)
%LOCALAPPDATA%\Stasis\icons\Cached app icons extracted from EXEs

Uninstalling

Use Windows Settings → Apps → Stasis → Uninstall. The uninstaller removes the application binaries and startup registry entry. It does not delete %LOCALAPPDATA%\Stasis\ — delete that folder manually for a full clean removal.

Stasis uses a dual-process architecture: a compiled Python telemetry engine that runs headlessly, and a React/Tauri desktop shell that provides the UI.

Security model

  • Credentials — Telegram token and chat ID are stored encrypted with Fernet symmetric encryption (AES-128-CBC + HMAC-SHA256).
  • Network exposure — Flask binds only to 127.0.0.1. No listener on external interfaces.
  • Single instance — Win32 named mutex Local\StasisSingleInstance prevents duplicate runs.
  • Danger operations — Destructive API endpoints require custom confirmation HTTP headers.

App categories (app_categories.json)

Every tracked executable is mapped to a main category and optional sub-category. URL rules override app rules when a browser is active.

{
  "apps": {
    "code.exe":     { "main": "productive",    "sub": "development" },
    "chrome.exe":   { "main": "neutral",        "sub": "browser"     },
    "discord.exe":  { "main": "communication",  "sub": "chat"        },
    "vlc.exe":      { "main": "entertainment",  "sub": "video"       }
  },
  "url_rules": {
    "github.com":        { "main": "productive",    "sub": "development" },
    "youtube.com/watch": { "main": "entertainment", "sub": "video"       },
    "mail.google.com":   { "main": "communication", "sub": "email"       }
  }
}

Wildcard patterns like *.github.com are supported in URL rules.

Ignored processes (ignored_apps.json)

Processes in this list are silently filtered from all reports. It is pre-populated with Windows system processes (e.g., dwm.exe, svchost.exe).

Database Schema

Stasis uses SQLite with WAL mode at %LOCALAPPDATA%\Stasis\data\stasis.db. The database has 6 tables covering raw telemetry, daily aggregates, limits, blocking state, settings, and optional file events.

Tables

activity_logs — Raw telemetry

CREATE TABLE activity_logs (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp       TEXT NOT NULL,      -- YYYY-MM-DD HH:MM:SS
    app_name        TEXT,               -- e.g. "chrome.exe"
    exe_path        TEXT,
    pid             INTEGER,
    window_title    TEXT,
    url             TEXT,               -- browsers only; NULL otherwise
    active_seconds  INTEGER DEFAULT 0,
    idle_seconds    INTEGER DEFAULT 0,
    keystrokes      INTEGER DEFAULT 0,
    clicks          INTEGER DEFAULT 0
);

daily_stats — Aggregated daily data

CREATE TABLE daily_stats (
    date            TEXT NOT NULL,
    app_name        TEXT NOT NULL,
    main_category   TEXT,
    sub_category    TEXT,
    active_seconds  INTEGER DEFAULT 0,
    idle_seconds    INTEGER DEFAULT 0,
    sessions        INTEGER DEFAULT 0,
    keystrokes      INTEGER DEFAULT 0,
    clicks          INTEGER DEFAULT 0,
    PRIMARY KEY (date, app_name, main_category)
);

app_limits — User-defined daily budgets

CREATE TABLE app_limits (
    id                   INTEGER PRIMARY KEY AUTOINCREMENT,
    app_name             TEXT UNIQUE NOT NULL,
    daily_limit_seconds  INTEGER NOT NULL,
    is_enabled           INTEGER DEFAULT 1,  -- 0=paused, 1=active
    created_at           TEXT,
    unblock_until        TEXT                -- ISO-8601 for temp unblocks
);

blocked_apps · settings · file_logs

  • blocked_apps (app_name PK, blocked_at) — Runtime cache of processes currently over their limit.
  • settings (key PK, value) — Key-value store for all settings and encrypted credentials.
  • file_logs (id, timestamp, action, file_path) — Optional file system event log.

SQLite pragmas

PRAGMA journal_mode = WAL;       -- concurrent reads during writes
PRAGMA synchronous  = NORMAL;    -- balanced speed / durability
PRAGMA busy_timeout = 10000;     -- wait up to 10 s if locked

Useful queries

-- Today's screen time by app
SELECT app_name, SUM(active_seconds) AS total
FROM daily_stats
WHERE date = date('now')
GROUP BY app_name
ORDER BY total DESC;

-- 14-day productivity trend
SELECT date,
    ROUND(100.0 *
        SUM(CASE WHEN main_category='productive'
                 THEN active_seconds ELSE 0 END)
        / MAX(SUM(active_seconds), 1), 1) AS pct
FROM daily_stats
GROUP BY date
ORDER BY date DESC LIMIT 14;

Telegram Integration

Stasis includes a private Telegram Bot integration that turns your phone into a remote control for your PC — no inbound ports required, works behind NAT and corporate firewalls.

Setup

  1. Open Telegram and message @BotFather → send /newbot → copy your Bot Token.
  2. Message @userinfobot to find your numeric Chat ID.
  3. In Stasis, open Settings → Telegram Integration.
  4. Enter both values, click Validate, then click Enable.

Both credentials are immediately encrypted with Fernet and stored only in the local SQLite database.

App Limits & Blocking

Set daily time budgets per application. Stasis automatically enforces them by terminating the process when the limit is reached — with an option to temporarily unblock for a meeting or exception.

How it works

  1. Go to the Limits tab and click Add Limit.
  2. Select an app (from activity history or the Windows registry).
  3. Set a daily limit in hours and minutes using the slider.
  4. BlockingService._limit_monitor checks usage every 15 seconds. When today's cumulative active_seconds exceeds the limit, the app is added to blocked_apps.
  5. BlockingService._process_guard scans running processes every 0.5 seconds and terminates any that are in blocked_apps.

Temporary unblock

Click Unblock for X minutes to override a block. This sets unblock_until = now + X minutes in the database. The blocking service respects this expiry timestamp and won't re-block until it passes.

Enable / Disable toggle

Toggle a limit off without deleting it — useful for weekends or vacations. is_enabled = 0 makes BlockingService skip that limit entirely.

Limits API

GET /limits/all All limits with status
POST /limits/set Create or update a limit
POST /limits/toggle Enable / disable a limit
POST /limits/unblock Temp unblock for N minutes
POST /limits/delete Remove a limit
GET /limits/blocked Currently blocked apps

Focus Scoring

The focus score (0–100) is computed live by the /api/focus endpoint using data from activity_logs. It rewards sustained deep work and penalises excessive task-switching and idle time.

Formula

Focus Score = min(100,
    deepWorkScore
  + flowBonus
  + engagementScore
  - switchPenalty
  - idlePenalty
)

Components

ComponentFormulaMax points
Deep work score(productiveSeconds / totalActive) × 100 × engagementMultiplier~65
Flow bonusNumber of unbroken ≥20-min productive streaks × 520
Engagement scoremin(15, (KPM / 35) × 15) where 35 KPM = baseline15
Switch penaltymax(0, (taskSwitches − 10) × 0.5)—
Idle penalty(idleSeconds / totalTime) × 20—

The API returns all raw values alongside the final score so the UI can display a detailed breakdown card per component.

Productivity %

A separate simpler metric: percentage of active time spent in productive-category apps, optionally weighted by keystroke intensity (KPM ratio). Available at /api/wellbeing.

Developer Guide

Everything you need to run Stasis locally, build the full application, and contribute to the project.

Run backend in development

# Create and activate virtual environment
python -m venv venv
venv\Scripts\activate

# Install dependencies
pip install -r requirements.txt

# Start the backend (API on http://127.0.0.1:7432)
python src/main.py

Run frontend in development

# Make sure the Python backend is running first
cd frontend
npm install       # first time only
npm run dev       # Vite dev server on http://localhost:5173
Tip: The Tauri shell is not needed for frontend development. A regular browser works fine since the frontend only needs the HTTP API.

Full production build

# From PowerShell in the project root
.\build.ps1 -Version 1.2.3

The build script:

  1. Syncs the version into tauri.conf.json, package.json, and Cargo.toml
  2. Generates Windows EXE version metadata
  3. Runs PyInstaller → dist/stasis-backend.exe
  4. Copies the backend EXE to frontend/src-tauri/bin/
  5. Runs npm install + npm run tauri:build

Output: frontend/src-tauri/target/release/bundle/nsis/Stasis-*.exe

Linting

cd frontend
npm run lint        # ESLint 9 with React + Hooks rules
npm run lint -- --fix
